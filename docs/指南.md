## 项目定位

这是一个**演示仓库**，不是业务项目。前端代码故意保持最小，重点在于把 GitHub 的协作流程跑通：

```
本地改代码 → push → 触发 CI → 提 PR → AI 自动 Review → 人工 Review → merge
```

## CI 原理与配置

### 什么是 CI（Continuous Integration）

每次有人 push 代码或提 PR，GitHub 自动运行一套检查脚本，防止"坏代码"进入主分支。

### 触发条件（`.github/workflows/ci.yml`）

```yaml
on:
  push:
    branches:
      - main
      - master
      - "feature/**"   # feature/ 开头的分支 push 时触发
  pull_request:        # 所有 PR 都触发
```

**触发逻辑**：只要满足上面任意一个条件，GitHub Actions 就会自动启动。

### 执行步骤

```
1. 拉取代码（checkout）
2. 安装 Node.js 20
3. npm ci（干净安装依赖）
4. npm run lint（ESLint 检查）
5. npm run build（Vite 构建）
```

任意一步失败 → CI 标红 → PR 无法合并（需配合 branch protection 设置）。

### 本地验证（推送前必跑）

```bash
npm run check   # = lint + build，等价于 CI 跑的内容
```

## CR 原理与配置

### 什么是 CR（Code Review）

开发者提 PR 后，其他人（或 AI）对代码变更进行审查，发现问题、讨论方案，确认无误后批准合并。

### GitHub 上的 CR 流程

```
1. 从 main 切出 feature 分支
2. 本地改代码，commit，push
3. 在 GitHub 上 New Pull Request（选 base: main）
4. 填写 PR 模板（改了什么、为什么、风险点）
5. Reviewer 查看 diff，留 comment 或 Approve
6. CI 和 AI Review 自动运行
7. 所有检查通过 + 至少 1 人 Approve → 可以 merge
```

### Branch Protection 配置（GitHub 仓库 Settings）

```
Settings → Branches → Add branch protection rule
Branch name pattern: main

勾选：
✅ Require a pull request before merging
  ✅ Require approvals（建议 1）
✅ Require status checks to pass before merging
  → 搜索并添加 "validate"（CI job 名）
✅ Require conversation resolution before merging
✅ Do not allow bypassing the above settings
```

设置完成后，直接 push 到 main 会被拒绝，必须走 PR 流程。

## AI Review 配置

本项目使用 **PR-Agent**（开源）作为 AI Review 方案，支持随时切换 AI 后端。

| 方案 | 内联行评论 | 自动描述 | 交互命令 | 自定义规则 | 费用 |
|------|-----------|---------|---------|-----------|------|
| **PR-Agent（当前）** | ✅ | ✅ | ✅ `/review` `/ask` | ✅ `.pr_agent.toml` | 免费（用 Gemini 额度） |
| Gemini Code Assist（已停用） | ✅ | ❌ | ✅ `/gemini` | ✅ styleguide.md | 免费但数据安全存疑 |

---

### PR-Agent 配置（当前方案）

**相关文件**：

| 文件 | 作用 |
|------|------|
| `.github/workflows/pr-agent.yml` | 触发入口，PR 打开/更新时自动运行 |
| `.pr_agent.toml` | 审查规则、模型配置、输出语言 |

**PR-Agent 做了什么**：
1. PR 打开时自动生成 PR 描述（摘要 + 改动说明）
2. 自动对 diff 进行 Code Review，输出中文评论，区分「必须修改」和「建议优化」
3. 自动打标签
4. 支持在 PR 评论里用命令与 AI 交互

**可用命令**（在 PR 评论里输入）：

```
/review          # 手动触发 review
/describe        # 重新生成 PR 描述
/improve         # 获取代码改进建议
/ask 你的问题    # 直接问 AI（例：/ask 这段逻辑为什么这样写？）
```

**定制审查规则**：编辑 `.pr_agent.toml` 中的 `extra_instructions`，push 后即生效。

---

### 切换 AI 后端

只需修改 `.github/workflows/pr-agent.yml` 中的两行环境变量：

**切换到 Claude（Anthropic）**：
1. GitHub Secrets 加 `ANTHROPIC_API_KEY`（在 console.anthropic.com 获取，按 token 计费，约 $0.014/PR）
2. 编辑 `pr-agent.yml`，注释掉 Gemini 的两行，取消注释 Claude 的两行

**切换到 GPT（OpenAI）**：
1. GitHub Secrets 加 `OPENAI_API_KEY`
2. 编辑 `pr-agent.yml`，切换到 GPT 那两行

三种后端的 review 质量对比：Claude ≈ GPT-4o > Gemini（但 Gemini 免费）。

## 前端 CR 打分机制

### 五维评分表

PR 模板里已内置，reviewer 在 GitHub 上直接填：

| 维度 | 说明 | 满分 |
|------|------|------|
| **功能正确性** | 逻辑是否正确，边界处理，有无明显 bug | 10 |
| **代码质量** | 可读性、命名规范、结构清晰度 | 10 |
| **性能** | 有无不必要的重渲染、大循环、资源浪费 | 10 |
| **安全性** | 有无 XSS、敏感数据泄露等风险 | 10 |
| **可维护性** | 是否易于后续修改和扩展，耦合度 | 10 |

**综合评分参考**：
- 45-50 分：直接合并（标签 `ai-approved`）
- 40-44 分：小改后合并（标签 `ai-low-risk`）
- 35-39 分：建议修改（标签 `ai-medium-risk`，workflow 标红）
- 34 分以下：需要讨论或重构（标签 `ai-high-risk`，workflow 标红）

### AI Review 也使用同样维度

Gemini（和预留的 Claude）的 prompt 已配置为按上述五个维度输出打分和建议，和人工 Review 表格对齐，方便对比。

### 示例：一个 PR 的完整 Review 流程

```
1. 开发者提 PR，填写 PR 模板（改了什么、风险点）
2. CI 自动运行 → 30s 内出结果（lint + build）
3. Gemini 自动 Review → 输出五维评分 + 改进建议
4. Reviewer 打开 PR，查看 diff 和 Gemini 评论
5. Reviewer 在 PR 模板表格里填自己的评分
6. 如有问题，在具体代码行留 comment
7. 开发者修改后 push，触发新一轮 CI + AI Review
8. Reviewer Approve → Merge
```

## 演示分支操作示例

```bash
# 1. 从 main 切出 feature 分支
git checkout main && git pull
git checkout -b feature/update-hero-text

# 2. 改一行代码（比如 src/main.js 里的标题文字）

# 3. 验证通过后提交
npm run check
git add src/main.js
git commit -m "feat: update hero section copy"

# 4. push 并提 PR
git push -u origin feature/update-hero-text
# → 在 GitHub 上 New Pull Request → 填模板 → 等 CI + AI Review
```
