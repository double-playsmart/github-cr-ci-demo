## 项目定位

这是一个**演示仓库**，不是业务项目。前端代码故意保持最小，重点在于把 GitHub 的协作流程跑通：

```
本地改代码 → push → 触发 CI → 提 PR → AI 自动 Review → 人工 Review → merge
```

## CI 原理与配置

### 什么是 CI（Continuous Integration）

每次有人 push 代码或提 PR，GitHub 自动运行一套检查脚本，防止"坏代码"进入主分支。

### 触发条件（`.github/workflows/ci.yml`）

```yaml
on:
  push:
    branches:
      - main
      - master
      - "feature/**"   # feature/ 开头的分支 push 时触发
  pull_request:        # 所有 PR 都触发
```

**触发逻辑**：只要满足上面任意一个条件，GitHub Actions 就会自动启动。

### 执行步骤

```
1. 拉取代码（checkout）
2. 安装 Node.js 20
3. npm ci（干净安装依赖）
4. npm run lint（ESLint 检查）
5. npm run build（Vite 构建）
```

任意一步失败 → CI 标红 → PR 无法合并（需配合 branch protection 设置）。

### 本地验证（推送前必跑）

```bash
npm run check   # = lint + build，等价于 CI 跑的内容
```

## CR 原理与配置

### 什么是 CR（Code Review）

开发者提 PR 后，其他人（或 AI）对代码变更进行审查，发现问题、讨论方案，确认无误后批准合并。

### GitHub 上的 CR 流程

```
1. 从 main 切出 feature 分支
2. 本地改代码，commit，push
3. 在 GitHub 上 New Pull Request（选 base: main）
4. 填写 PR 模板（改了什么、为什么、风险点）
5. Reviewer 查看 diff，留 comment 或 Approve
6. CI 和 AI Review 自动运行
7. 所有检查通过 + 至少 1 人 Approve → 可以 merge
```

### Branch Protection 配置（GitHub 仓库 Settings）

```
Settings → Branches → Add branch protection rule
Branch name pattern: main

勾选：
✅ Require a pull request before merging
  ✅ Require approvals（建议 1）
✅ Require status checks to pass before merging
  → 搜索并添加 "validate"（CI job 名）
✅ Require conversation resolution before merging
✅ Do not allow bypassing the above settings
```

设置完成后，直接 push 到 main 会被拒绝，必须走 PR 流程。

## AI Review 配置

本项目同时支持两套 AI Review 方案，可单独使用或并用。

| 方案 | 内联行评论 | 打分 | 阻断合并 | 自定义规则 | 费用 |
|------|-----------|------|---------|-----------|------|
| Gemini Code Assist（GitHub App） | ✅ | ❌ | ❌ | ✅ styleguide.md | 免费（33 PR/天） |
| Actions + Gemini API | ❌ | ✅ | ✅ | ✅ CUSTOM_RULES | 免费（250 次/天） |

**推荐并用**：Gemini Code Assist 负责逐行评论，Actions 负责打分和阻断合并。

---

### 方案一：Gemini Code Assist（GitHub App）

**安装**：[github.com/marketplace/gemini-code-assist](https://github.com/marketplace/gemini-code-assist) → Install → 选择仓库

**配置文件**（仓库根目录 `.gemini/` 目录）：

| 文件 | 作用 |
|------|------|
| `.gemini/config.yaml` | 控制触发行为、严重级别阈值、忽略文件 |
| `.gemini/styleguide.md` | 自然语言审查规则，Gemini 会遵守 |

修改这两个文件后 push 即生效，无需重新配置。

**免费版限制**：
- 33 PR/天（约 1000 PR/月）
- 只在 PR 打开时触发，不可自定义触发时机
- 不能作为 required status check 阻断合并

**可用命令**（在 PR 评论里输入）：
```
/gemini review     # 手动触发审查
/gemini summary    # 生成 PR 摘要
```

---

### 方案二：Actions + Gemini API（当前方案）

**Gemini API 免费额度**（AI Studio，与 Gemini Pro 订阅无关，独立计费）：

| 模型 | RPM | RPD（每天请求上限） |
|------|-----|-------------------|
| gemini-2.5-flash | 10 | 250 |
| gemini-2.5-flash-lite | 15 | **1000**（推荐 CI 场景） |
| gemini-2.5-pro | 5 | 100 |

文档：[ai.google.dev/gemini-api/docs/rate-limits](https://ai.google.dev/gemini-api/docs/rate-limits)

**相关文件**：
- `.github/workflows/claude-review.yml`：触发入口，PR 打开/更新/重新打开时运行
- `.github/scripts/gemini_review.py`：实际执行逻辑

**gemini_review.py 做了什么**：
1. 自动探测当前 API key 可用的 Gemini 模型（优先 flash 系列）
2. 读取 workflow 生成的 `diff.txt`，用游戏前端专项 prompt 发给 Gemini
3. 按五维度（功能正确性/代码质量/性能/安全性/可维护性）各评 0-10 分
4. 将评分格式化为 Markdown 表格，用 `gh pr comment` 贴到 PR 评论区
5. 根据总分自动给 PR 打标签：`ai-approved`（≥45）/ `ai-low-risk`（40-44）/ `ai-medium-risk`（35-39）/ `ai-high-risk`（<35）
6. 总分 ≥ 40/50 → exit 0（通过），< 40 → exit 1（workflow 标红，可阻断合并）

**配置步骤**：
1. 去 [aistudio.google.com](https://aistudio.google.com) 登录 Google 账号
2. 左侧「Get API key」→ Create API key → 复制
3. GitHub 仓库 → Settings → Secrets and variables → Actions
4. New repository secret，名称填 `GEMINI_API_KEY`，值粘贴 key

配置完成后，提 PR 约 30 秒内 Gemini 会在 PR 评论区自动输出 Review。

### 切换到 Claude（团队目标）

Claude 需要 Anthropic API Key（独立于 Claude Max 订阅，需在 console.anthropic.com 充值）。

切换步骤（已预留注释）：
1. 在 GitHub Secrets 中添加 `ANTHROPIC_API_KEY`
2. 打开 `.github/workflows/claude-review.yml`
3. 注释掉 `gemini-review` job
4. 取消注释文件底部的 `claude-review` job

## 前端 CR 打分机制

### 五维评分表

PR 模板里已内置，reviewer 在 GitHub 上直接填：

| 维度 | 说明 | 满分 |
|------|------|------|
| **功能正确性** | 逻辑是否正确，边界处理，有无明显 bug | 10 |
| **代码质量** | 可读性、命名规范、结构清晰度 | 10 |
| **性能** | 有无不必要的重渲染、大循环、资源浪费 | 10 |
| **安全性** | 有无 XSS、敏感数据泄露等风险 | 10 |
| **可维护性** | 是否易于后续修改和扩展，耦合度 | 10 |

**综合评分参考**：
- 45-50 分：直接合并（标签 `ai-approved`）
- 40-44 分：小改后合并（标签 `ai-low-risk`）
- 35-39 分：建议修改（标签 `ai-medium-risk`，workflow 标红）
- 34 分以下：需要讨论或重构（标签 `ai-high-risk`，workflow 标红）

### AI Review 也使用同样维度

Gemini（和预留的 Claude）的 prompt 已配置为按上述五个维度输出打分和建议，和人工 Review 表格对齐，方便对比。

### 示例：一个 PR 的完整 Review 流程

```
1. 开发者提 PR，填写 PR 模板（改了什么、风险点）
2. CI 自动运行 → 30s 内出结果（lint + build）
3. Gemini 自动 Review → 输出五维评分 + 改进建议
4. Reviewer 打开 PR，查看 diff 和 Gemini 评论
5. Reviewer 在 PR 模板表格里填自己的评分
6. 如有问题，在具体代码行留 comment
7. 开发者修改后 push，触发新一轮 CI + AI Review
8. Reviewer Approve → Merge
```

## 演示分支操作示例

```bash
# 1. 从 main 切出 feature 分支
git checkout main && git pull
git checkout -b feature/update-hero-text

# 2. 改一行代码（比如 src/main.js 里的标题文字）

# 3. 验证通过后提交
npm run check
git add src/main.js
git commit -m "feat: update hero section copy"

# 4. push 并提 PR
git push -u origin feature/update-hero-text
# → 在 GitHub 上 New Pull Request → 填模板 → 等 CI + AI Review
```
