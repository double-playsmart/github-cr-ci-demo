# GitHub CR / CI 工作流汇报总结

> 面向游戏组前端团队的工程协作流程研究与落地演示

---

## 一、我们要解决什么问题

游戏组当前前端项目（games2、game-core、game-sdk 等）体量大、多人并行开发，常见痛点：

- 代码改动没有统一的 review 入口，靠口头沟通
- 合并出问题才发现，没有自动化前置拦截
- review 质量参差不齐，没有量化标准
- AI 工具人手一份但各自为阵，没有接入团队流程

**这个演示仓库的目标**：把上面这些问题的解法跑通，形成可复制的标准流程。

---

## 二、实现目标

| 目标 | 状态 |
|------|------|
| 本地改代码 → push → 自动触发检查 | ✅ |
| PR 流程规范化（模板 + 审批） | ✅ |
| CI 自动 lint + build，失败阻断合并 | ✅ |
| AI 自动对 PR 进行 Code Review 并评论 | ✅ |
| AI 评分低于阈值时阻断合并 | ✅ |
| 文档可被 Claude / Gemini 等 AI 续接，跨会话协作 | ✅ |

## 三、整体链路原理

```
开发者本地
  │
  ├─ git checkout -b feature/xxx     # 从 main 切出分支
  ├─ 改代码
  ├─ npm run check                   # 本地验证（lint + build）
  └─ git push
         │
         ▼
    GitHub 收到 push
         │
         ├─ 触发 CI（ci.yml）
         │    └─ lint + build → 通过/失败
         │
         └─ 开发者提 Pull Request
                  │
                  ├─ 触发 AI Review（claude-review.yml）
                  │    ├─ 读取 PR diff
                  │    ├─ 调用 Gemini API
                  │    ├─ 输出五维评分 + 建议（发 PR 评论）
                  │    └─ 评分 < 40/50 → job 失败 → 阻断合并
                  │
                  ├─ Reviewer 人工 review + Approve
                  │
                  └─ 所有检查通过 → 可以 Merge 进 main
```

## 五、核心配置说明

### CI（`.github/workflows/ci.yml`）

```yaml
on:
  push:
    branches: [main, "feature/**"]
  pull_request:              # 所有 PR 都触发

jobs:
  validate:
    steps:
      - npm ci
      - npm run lint         # ESLint 检查
      - npm run build        # Vite 构建
```

**关键点**：job 名 `validate` 需要在 branch protection 里设为 required，失败时阻断合并。

---

### AI Review（`.github/workflows/claude-review.yml`）

```yaml
on:
  pull_request:
    types: [opened, synchronize, reopened]
```

**执行逻辑**：
1. 拉取 PR diff（与 base branch 的差异）
2. 用 Python 调用 Gemini API，发送 diff + 评审 prompt
3. Gemini 返回五维评分 + `TOTAL: XX/50`
4. 解析总分：≥ 40 → job 成功；< 40 → job 失败
5. 自动在 PR 下面发评论（包含评分详情和建议）

**切换到 Claude**：在 GitHub Secrets 加 `ANTHROPIC_API_KEY`，取消 workflow 文件底部注释即可。

---

### Branch Protection（GitHub Settings）

```
main 分支保护规则：
  ✅ 必须通过 PR 才能合并（至少 1 人 Approve）
  ✅ 必须通过 status checks：validate + gemini-review
  ✅ 禁止直接 push 到 main
```

## 六、AI 五维评分标准

| 维度 | 说明 | 满分 |
|------|------|------|
| 功能正确性 | 逻辑是否正确，有无明显 bug | 10 |
| 代码质量 | 可读性、命名规范、结构清晰度 | 10 |
| 性能 | 有无不必要的重渲染、大循环、资源浪费 | 10 |
| 安全性 | 有无 XSS、敏感数据泄露等风险 | 10 |
| 可维护性 | 是否易于后续修改和扩展 | 10 |

**阈值**：总分 ≥ 40/50 通过，< 40 自动阻断合并。

## 七、结合游戏组实际场景

### 当前技术栈映射

| 演示仓库 | 游戏组对应 |
|---------|-----------|
| Vite + 原生 JS | games2（SolidJS + Vite）|
| ESLint lint | tsc --noEmit + ESLint |
| npm run build | pnpm build |
| 单仓库 | games2 / game-core / game-sdk 多仓库 |

### 游戏组落地建议

**第一步（低成本试点）**：在 games2 或 game-sdk 的一个子包里加上 ci.yml + claude-review.yml，先跑通链路，验证团队接受度。

**第二步（扩展 CI）**：在 `npm run lint` 基础上加 `pnpm tsc --noEmit`，把类型检查也纳入 CI 门禁。游戏组代码改动频繁，类型错误是高频问题。

**第三步（调整 AI Review prompt）**：针对游戏组特点补充评审重点：
- FSM 状态转换逻辑是否正确
- Provider 接口变更是否向后兼容
- 游戏赔率计算是否有精度问题
- WebSocket 消息处理是否有竞态条件

**第四步（Claude 替换 Gemini）**：团队统一使用 Anthropic API Key，切换到 Claude，评审质量更高，且可以用 `claude-code-action` 做更复杂的分析（读源码、跑类型检查等）。

### AI 跨工具协作（已实现）

`docs/ai-context.md` 存储了项目背景和约定，Claude Code、Gemini、ChatGPT 等任何 AI 都能读取后无缝接手任务，不依赖单一工具或个人的上下文记忆。这套模式可以直接平移到 games2 的 `internal/` 目录体系。

## 八、今天演示的核心结论

1. **CI 不是"高级配置"**，10 分钟就能给任何项目加上自动检查门禁
2. **AI Review 不是"手动贴代码给 AI 看"**，而是真正接入 PR 流程、自动触发、自动评论
3. **评分阻断合并** 把 AI 意见变成了有约束力的流程节点，不是摆设
4. **Gemini 免费可用**，今天就能跑通；Claude 是团队目标，切换成本极低
5. **文档即上下文**，结构化文档让 AI 成为真正的团队协作者，而不只是个人助手
