# GitHub CR / CI 工作流汇报总结

> 面向游戏组前端团队的工程协作流程研究与落地演示

---

## 一、我们要解决什么问题

游戏组当前前端项目（games2、game-core、game-sdk 等）体量大、多人并行开发，常见痛点：

- 代码改动没有统一的 review 入口，靠口头沟通
- 合并出问题才发现，没有自动化前置拦截
- review 质量参差不齐，没有量化标准
- AI 工具人手一份但各自为阵，没有接入团队流程

**这个演示仓库的目标**：把上面这些问题的解法跑通，形成可复制的标准流程。

## 三、整体链路原理

```
开发者本地
  │
  ├─ git checkout -b feature/xxx     # 从 main 切出分支
  ├─ 改代码
  ├─ npm run check                   # 本地验证（lint + build）
  └─ git push → 提 Pull Request
         │
         ▼
    GitHub 收到 PR
         │
         ├─ 触发 CI（ci.yml）
         │    ├─ lint + build → 通过/失败
         │    └─ reviewdog → ESLint 错误显示为 PR inline comment
         │
         ├─ 触发 PR-Agent（pr-agent.yml）
         │    ├─ 自动生成 PR 描述（类型 + 改动说明 + 文件列表）
         │    ├─ 输出中文 Review 评论（安全/性能/代码质量）
         │    ├─ 评估 review 所需精力（1-5）
         │    └─ 支持 /ask /improve /describe 等交互命令
         │
         ├─ Reviewer 人工 review + Approve
         │
         └─ 所有检查通过 → 可以 Merge 进 main
```

## 五、核心配置说明

### CI（`.github/workflows/ci.yml`）

```yaml
on:
  push:
    branches: [main, "feature/**"]
  pull_request:

jobs:
  validate:
    steps:
      - npm ci
      - npm run lint              # ESLint 检查（失败阻断合并）
      - reviewdog/action-eslint   # ESLint 错误显示为 PR inline comment
      - npm run build             # Vite 构建
```

**关键点**：job 名 `validate` 在 branch protection 里设为 required，失败时阻断合并。

---

### AI Review（`pr-agent.yml` + `.pr_agent.toml`）

```yaml
on:
  pull_request:
    types: [opened, reopened, ready_for_review]
  issue_comment:   # 支持 /review /ask /improve 等命令
```

**执行逻辑**：
1. PR 打开 → PR-Agent 读取 diff 和 `.pr_agent.toml` 配置
2. 调用 Gemini API（flash-lite，1000次/天免费）
3. 自动生成 PR 描述，发出中文 Review 评论
4. 评论包含：精力评估（1-5）、安全问题分析、重点审查项

**切换 AI 后端**：编辑 `pr-agent.yml` 中两行环境变量，支持 Claude / GPT / Gemini 无缝切换。

**可用交互命令**（在 PR 评论里输入）：

| 命令 | 功能 |
|------|------|
| `/review` | 手动触发 review |
| `/improve` | 给出具体代码改进建议（含可 commit 的 diff） |
| `/ask 问题` | 针对这个 PR 问 AI 任何问题 |
| `/describe` | 重新生成 PR 描述 |

---

### Branch Protection（GitHub Settings）

```
main 分支保护规则：
  ✅ 必须通过 PR 才能合并（至少 1 人 Approve）
  ✅ 必须通过 status checks：validate（CI）
  ✅ 禁止直接 push 到 main
```

> 注：GitHub 免费个人账号的私有仓库不强制执行 ruleset，公开仓库或 Team 账号可完整生效。

## 六、AI Review 方案选型结论

经过调研对比，最终选择 **PR-Agent（开源自托管）** 作为当前方案：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| **PR-Agent（当前）** | 开源免费、多模型、功能全 | 有已知安全漏洞（prompt injection） | ✅ 当前使用 |
| CodeRabbit | 用户最多、开箱即用 | 2025年1月 RCE 漏洞，不推荐 | ❌ |
| claude-code-action | 官方出品、质量高 | 锁定 Claude，需付费 | 备选（团队确定用 Claude 时） |
| 自研脚本 | 完全可控 | 功能单一，维护成本高 | 已替换 |

**数据安全说明**：PR-Agent 自托管模式下，代码 diff 只发给你配置的 AI API（Gemini/Claude），不经过任何第三方中间服务器。

## 七、结合游戏组实际场景

### 当前技术栈映射

| 演示仓库 | 游戏组对应 |
|---------|-----------|
| Vite + 原生 JS | games2（SolidJS + Vite）|
| ESLint lint | tsc --noEmit + ESLint |
| npm run build | pnpm build |
| 单仓库 | games2 / game-core / game-sdk 多仓库 |

### 游戏组落地建议

**第一步（低成本试点）**：在 games2 或 game-sdk 的一个子包里加上 `ci.yml` + `pr-agent.yml`，先跑通链路，验证团队接受度。

**第二步（扩展 CI）**：在 lint 基础上加 `pnpm tsc --noEmit`，把类型检查纳入 CI 门禁。游戏组代码改动频繁，类型错误是高频问题。

**第三步（定制 AI 审查规则）**：编辑 `.pr_agent.toml` 中的 `extra_instructions`，针对游戏组特点补充：
- FSM 状态转换逻辑是否正确
- Provider 接口变更是否向后兼容
- 游戏赔率计算浮点精度
- WebSocket 竞态条件

**第四步（切换 Claude）**：在 `pr-agent.yml` 中改两行环境变量即可，无需重写任何配置。

## 八、核心结论

1. **CI 不是"高级配置"**，10 分钟就能给任何项目加上自动检查门禁
2. **AI Review 不是"手动贴代码给 AI 看"**，而是真正接入 PR 流程、自动触发、自动评论
3. **PR-Agent 是开源成熟方案**，不锁定任何 AI 厂商，现在用 Gemini（免费），随时切 Claude
4. **定制化靠配置文件**，改 `.pr_agent.toml` 的 `extra_instructions` 即可调整审查重点，push 后生效
5. **文档即上下文**，结构化文档让 AI 成为真正的团队协作者，而不只是个人助手
